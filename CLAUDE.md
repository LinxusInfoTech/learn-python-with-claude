## How to run sessions in this repo (IMPORTANT — read every time)
1. Read `ROADMAP.md` in this directory — it is the source of truth for what's done, in progress, and next.
2. Find the first task marked `[ ]` (not done) in the current phase. That is today's task.
3. Teach hands-on first: write/edit a small real script together, explain concepts *through* the code, not as a lecture wall before it.
4. Keep sessions tight and concrete — one script, one concept cluster, working output by the end.
5. When a task is completed, update `ROADMAP.md`: check it off `[x]` and add a one-line note on what was actually built/learned (date included).
6. If Sachin seems to be losing interest in a topic, it's fine to jump ahead to a more "interesting" adjacent task in the roadmap and circle back later — flexibility over rigid sequence, but always update the roadmap to reflect what actually happened.
7. Do NOT touch or reference the InfraSync codebase from this project unless Sachin explicitly asks to bridge the two (e.g., building an MCP server against InfraSync's Mongo).

## Switching back to InfraSync work
When Sachin says things like "switch back to infrasync" / "let's go back to fable5 and infrasync work" / "back to work":
- Stop the lab session, do NOT keep pushing lab tasks.
- Note in `ROADMAP.md` (one line, dated) where the session left off if not already saved.
- Acknowledge the switch and wait for Sachin's next instruction in the InfraSync context — do not carry lab exercise state into InfraSync work.
